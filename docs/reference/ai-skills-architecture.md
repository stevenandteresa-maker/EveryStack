## 1. The Problem

When an EveryStack user says "build me an automation that emails clients when their invoice is overdue," the AI currently has to:

1. Discover the workspace schema via SDS (what tables, fields, links exist)
2. Figure out what an automation IS (trigger types, action types, configuration shapes)
3. Figure out what valid combinations look like (which triggers support which configs)
4. Figure out the merge tag syntax for dynamic values
5. Actually build the automation

Steps 1 and 5 are productive work. Steps 2–4 are orientation — the AI re-learning EveryStack's own platform every single session.

For external integrations the problem is worse. When the AI needs to configure a webhook to HubSpot, or query Google Analytics data through MCP, or help a user build a cross-platform automation involving Stripe and Slack, the orientation cost multiplies — the AI must discover tool schemas, learn API conventions, understand response shapes, and map external data to EveryStack's internal model. With MCP specifically, tool discovery alone can burn significant tokens before any real work begins.

Runtime skills eliminate this waste by pre-loading curated knowledge into the AI's context, so it spends its token budget on productive work instead of orientation.

---

## 2. What Already Exists (and Why It's Not Enough)

**Schema Descriptor Service (SDS):** Gives the AI the user's data model — tables, fields, types, links, cardinality. Essential but only half the picture. Tells the AI WHAT data exists, not HOW the platform's features work or how external integrations behave.

**Prompt Registry with Versioned Templates:** Each AI feature has a prompt template with system instructions, output schemas, and few-shot examples. Tightly coupled to one task — no shared, reusable knowledge layer.

**Capability-Based Routing:** The fast/standard/advanced tier system is solid. Missing a step between intent classification and task execution where the system loads the right feature knowledge.

**Context Builder:** Assembles prompts from schema data. Has no mechanism to load feature-specific or integration-specific reference material alongside that schema data.

**MCP Architecture:** EveryStack as MCP Server (post-MVP — exposing tools to external AI) and MCP Client (post-MVP — consuming external services). The MCP layer is specced but has no knowledge pre-loading strategy — every session starts with full tool discovery.

**What's missing:** A knowledge layer between schema context and prompt templates that encodes how features work, how integrations behave, and how this specific workspace uses them.

## 3. The Three-Tier Skill Model

Skills operate at three tiers, each serving a different kind of knowledge:

### Tier 1 — Platform Skills (static, human-written)

Encode how EveryStack's features work. One skill per major feature domain. Content doesn't change per workspace — automations have 6 triggers whether the user has 3 tables or 300.

**Examples:** automation-builder.md, portal-builder.md, document-templates.md, record-management.md

**Authored by:** Platform team (or Skill Maintenance Agent drafts, human approves)
**Updated when:** Features change (new trigger types, new action types, new field types)
**Stored in:** `packages/shared/ai/skills/documents/` — version-controlled with the codebase

### Tier 2 — Workspace Context Skills (dynamic, generated per tenant)

Encode how a specific workspace uses the platform and its integrations. Generated from the tenant's actual configuration data, cached like SDS output.

**Examples:**
- Which tables store imported GA data, how fields map to metrics, sync schedule
- The user's most common automation patterns (15 automations, all using Record Created → Send Email)
- Which Cross-Links connect external data to internal tables
- Integration connection inventory (which MCP servers are connected, which API keys are configured)

**Generated by:** Workspace Usage Descriptor — a companion module to SDS that queries tenant configuration
**Updated when:** Tenant configuration changes (new integrations, new automations, schema mutations)
**Stored in:** Redis cache, same pattern as SDS: `skills:workspace:{workspaceId}:{configHash}`

### Tier 3 — Behavioral Skills (learned over time)

Encode patterns observed from actual usage. What reports this user runs regularly, what portal layouts they prefer, what automation structures they build most often.

**Examples:**
- "User runs a weekly GA report: last 7 days, sessions by source/medium, compared to previous period, exported to Weekly Marketing table"
- "User's portal configurations always include an editable Status field and a read-only Invoice Total"
- "User's automations always include a Send Email action with the company logo header template"

**Generated by:** Usage pattern analysis from `ai_usage_log` — scheduled aggregation job
**Updated when:** Weekly aggregation detects new dominant patterns
**Stored in:** Redis cache, keyed per user within workspace

### How the Three Tiers Compose

When the AI handles a request, the Context Builder loads skills from all three tiers:

```
User: "Run my weekly marketing report"

Tier 1 (Platform):    report-charts.md — chart types, aggregation functions, data binding modes
Tier 1 (Integration): google-analytics.md — GA MCP tools, query patterns, metric definitions
Tier 2 (Workspace):   This workspace's GA table mapping, field-to-metric correspondence, sync schedule
Tier 3 (Behavioral):  This user runs this report every Monday with these specific parameters

Result: AI skips all orientation, goes straight to execution with correct parameters
```

**Key interfaces:**

```typescript
interface SkillDocument {
  id: string;                     // 'automation-builder', 'google-analytics', etc.
  version: number;
  domain: AITaskType[];           // Which task types trigger this skill
  skillTier: 'platform' | 'integration';  // Tier 1 subcategory
  content: Record<SkillCondensationLevel, string>;  // full / standard / minimal

  // Integration skills only:

  id: string;

  level: SkillCondensationLevel;
  content: string;
  tokenEstimate: number;

```

Each skill has three pre-written condensation levels because feature knowledge doesn't condense well algorithmically — a human decides which facts are essential at each budget tier.

### Context Builder Enhancement

```
Current:
  User intent → Prompt template → [SDS schema] → LLM

Proposed:
  User intent → Intent classifier (fast tier) → resolves task type + relevant skills
    → Skill loader picks skills from all tiers for that task type
    → Token budget allocator divides budget: skills → schema → conversation
    → Skills loaded at appropriate condensation levels
    → SDS schema loaded at appropriate condensation level
    → Workspace + behavioral context loaded if available
    → All injected into prompt template → LLM
```

**Change to Context Builder:** Add a `loadSkills(taskType, skillIds, tokenBudget)` step after intent classification and before prompt assembly.

**Change to Prompt Registry:** Add optional `requiredSkills: string[]` field to `PromptTemplate`. When present, the Context Builder loads those specific skills. When absent, it infers from task type.

### Token Budget Allocator

Extends the SDS Token Budget Estimator into a shared budget across four consumers:

| Consumer | Priority | Budget Strategy |
|----------|----------|-----------------|
| Platform + integration skills | First | Fixed allocation based on condensation level. Small (500–2,000 tokens) but extremely high value per token. |
| Workspace context skill | Second | Fixed allocation, typically 200–500 tokens. Generated to fit budget. |
| SDS schema | Third | Remainder after skill allocations, using SDS condensation levels. |
| Conversation history / few-shot / behavioral | Fourth | Remainder after all above. |

### Skill-Aware Intent Classification

```typescript
interface SkillAwareIntent {
  taskType: AITaskType;
  confidence: number;
  relevantSkills: string[];                // ['automation-builder', 'stripe'] — from any tier
  complexityEstimate: 'simple' | 'moderate' | 'complex';
  involvedIntegrations?: string[];         // ['google-analytics', 'hubspot'] — triggers integration skills
}
```

The `involvedIntegrations` field lets the intent classifier signal that external platform knowledge is needed. The skill loader then includes integration skills alongside platform skills.

## 5. MCP Integration Skills

Covers The Token Problem with MCP, Skill Structure for Integrations, Version Awareness for External Skills, Adding a New Integration.

### The Token Problem with MCP

When the AI connects to an external service via MCP, it must discover available tools, read their schemas, understand parameter semantics, learn response shapes, and figure out common query patterns — all before doing productive work. This discovery repeats every session because MCP is stateless.

Integration skills eliminate this by pre-loading curated knowledge about each MCP server.

### Skill Structure for Integrations

Each integration skill covers:

- **Available MCP tools** — the specific endpoints the MCP server exposes, what each does, which parameters are required vs optional
- **Common query patterns** — the 5–10 most useful queries users actually run (sessions by source, revenue by product, contacts by lifecycle stage)
- **Response shapes** — what the data looks like when it comes back, so the AI can plan how to present or store it
- **Rate limits and pagination** — constraints the AI needs to respect
- **EveryStack mapping conventions** — how this platform's data typically maps to EveryStack field types (GA sessions → number field, traffic source → single_select field)
- **Version awareness** — which API version / MCP server version this skill was validated against

### Version Awareness for External Skills

External platforms update their APIs independently. Integration skills include metadata that enables drift detection:

```yaml
---
id: google-analytics
version: 3
domain:
  - conversation
  - summarize
skillTier: integration
mcpServerUrl: "https://ga.mcp.example.com/sse"
lastVerified: "2026-03-01"
externalApiVersion: "GA4 Data API v1"
tokenEstimates:
  full: 1100
  standard: 450
  minimal: 200
---
```

The `lastVerified` date and `externalApiVersion` enable automated drift detection — when evals fail against a live integration, the system knows the skill is stale.

### Adding a New Integration

When a new MCP integration is added to EveryStack, the process is:

1. Connect the MCP server
2. Explore its tool set (one-time discovery)
3. Write the Tier 1 integration skill document (tools, patterns, response shapes, mapping conventions)
4. Write eval cases for common queries
5. Run evals → validate skill quality
6. Ship the skill alongside the integration

## 6. API Knowledge Skills

The AI doesn't make direct HTTP calls to external services — EveryStack routes external communication through three controlled channels: sync adapters (data import/export), MCP (AI tool access), and webhooks (automation-triggered outbound calls). But the AI frequently needs to *understand* external APIs even when it's not calling them directly.

### Where API Knowledge Matters

**Webhook configuration:** When a user says "set up a webhook to update our HubSpot contacts when a record changes," the AI needs to know HubSpot's contact update endpoint, accepted fields, auth requirements, and payload format — not to call it, but to configure the Send Webhook action correctly.

**Data translation explanations:** When the sync engine pulls data from Airtable and the user asks "why are my attachment fields showing as URLs," the AI needs to understand Airtable's API response shape to explain the translation.

**Multi-step webhook workflows:** When a user builds an automation chaining Stripe (create invoice) → Slack (notify team) → HubSpot (update deal stage), the AI needs all three APIs' payload shapes to configure each webhook action correctly.

### API Skills vs MCP Skills

These are not separate — they're the same skill documents. An integration skill for HubSpot covers both the MCP tools (if connected) and the webhook/API knowledge (always useful). The MCP-specific sections (tool names, parameter schemas) are in the `full` condensation level. The API knowledge sections (endpoint URLs, payload shapes, auth patterns) are in all three levels because they're useful even without MCP.

This means a user who hasn't connected the HubSpot MCP server still gets value from the skill when configuring webhook automations. The skill degrades gracefully — less capability (no live tool calls) but the reference knowledge remains.

## 7. Workspace Usage Descriptor (Tier 2 Generator)

Covers Purpose, What It Produces, Architecture, Phase Placement.

### Purpose

Companion module to SDS. While SDS provides schema context (what tables and fields exist), the Workspace Usage Descriptor provides behavioral context (how the workspace uses features and integrations).

### What It Produces

A condensed JSON document (similar to SDS output) that covers:

```typescript
interface WorkspaceUsageDescriptor {
  workspaceId: string;

  // Integration inventory
  connectedIntegrations: Array<{
    platform: string;              // 'google-analytics', 'hubspot', etc.
    mcpConnected: boolean;         // Is the MCP server active?
    syncAdapterConnected: boolean; // Is there a sync adapter?
    targetTables: string[];        // Which EveryStack tables receive this data
    fieldMappings: Array<{         // How external fields map to EveryStack fields
      externalField: string;
      everyStackField: string;
      fieldType: string;
    }>;
    syncSchedule?: string;         // 'every 15 min', 'hourly', 'manual'
  }>;

  // Automation patterns
  automationSummary: {
    totalCount: number;
    triggerDistribution: Record<string, number>;  // { 'record_created': 8, 'scheduled': 3, ... }
    actionDistribution: Record<string, number>;
    commonPatterns: Array<{
      trigger: string;
      actions: string[];
      frequency: number;           // How many automations follow this pattern
    }>;
  };

  // Portal patterns
  portalSummary: {
    totalCount: number;
    commonFieldConfigs: string[];  // Most frequently included fields across portals
    authPreference: 'magic_link' | 'password' | 'mixed';
  };

  // Document template patterns
  documentSummary: {
    totalCount: number;
    commonMergeTags: string[];     // Most frequently used merge tags
  };
}
```

### Architecture

```
┌──────────────────────────────────────────────┐
│              Context Builder                   │
│  loadSkills() + loadSchema() + loadUsage()     │
└──────┬──────────────┬────────────┬────────────┘
       │              │            │
       ▼              ▼            ▼
  Skill Registry   SDS          Workspace Usage Descriptor
  (Tier 1 skills)  (schema)    (Tier 2 context)
       │              │            │
       │              │            ├── queries automations table
       │              │            ├── queries portals table
       │              │            ├── queries integrations config
       │              │            └── queries document_templates
       │              │
       │              └── queries field_definitions, cross_links
       └── reads static markdown files
```

**Caching:** Same pattern as SDS. Two-tier cache (workspace-level + user-filtered). Invalidated on configuration changes (new automation, new portal, new integration). Redis key: `wud:{workspaceId}:{configHash}`. TTL: 10 minutes.

### Phase Placement

The Workspace Usage Descriptor is a Phase 5 or early post-MVP deliverable. It depends on automations, portals, and integrations being built first (there's nothing to describe if no features have been configured). The `WorkspaceUsageDescriptor` interface can be defined in the Foundation skeleton alongside the skill types — the implementation ships when the features it describes exist.

## 8. Monitoring & Observability

Covers Where You Monitor: Platform Owner Console, Data Requirements.
Touches `ai_usage_log` tables.

### Where You Monitor: Platform Owner Console

Skill performance monitoring lives in Settings → AI → Skill Performance within the Platform Owner Console. Three panels:

**Skill Health Dashboard**

Per-skill status indicators (green/yellow/red) based on composite health score:

| Signal | Weight | Source |
|--------|--------|--------|
| Eval suite pass rate | 40% | Scheduled eval runs |
| User acceptance rate (1 - regeneration rate) | 30% | `ai_usage_log` with `skillContext` |
| Token efficiency (productive tokens / total tokens) | 20% | `ai_usage_log` token counts |
| Trend direction (improving / stable / declining) | 10% | Week-over-week delta of above metrics |

**Thresholds:** Green ≥ 85% composite. Yellow 70–84%. Red < 70%.

Integration skills additionally show: `lastVerified` date, eval drift status (pass/warn/fail), external API version tracked.

**Failure Pattern Explorer**

The 10 most common failure cases per skill, grouped by error type:

- Invalid configuration generated (wrong JSONB shape, nonexistent option)
- User regenerated (accepted on 2nd or 3rd attempt — skill got close but not right)
- Hallucinated feature (AI referenced something that doesn't exist — missing skill coverage)
- Integration drift (external API behavior doesn't match skill description)

Each entry shows anonymized examples: the user's intent, what the AI produced, what the user changed or regenerated. This is the primary input for skill improvement — whether manual or agent-driven.

**Token Efficiency Panel**

Per-skill and per-task-type breakdown of token usage:

- Orientation tokens (tokens spent before the AI begins productive work)
- Productive tokens (tokens spent on actual output generation)
- Skill load tokens (the cost of loading the skill itself)
- Net savings (estimated orientation tokens avoided by having the skill)

This panel answers the question "are skills actually saving tokens?" and identifies task types where orientation costs remain high — suggesting a skill gap.

### Data Requirements

All monitoring data comes from existing infrastructure with one addition:

**Existing:** `ai_usage_log` already tracks feature, model, tokens, cost, duration, and user action (accept/reject/regenerate).

**New column:** `skill_context JSONB` on `ai_usage_log` — records which skills were loaded, at what condensation level, and their token cost. Schema:

```typescript
interface SkillContextLog {
  skills: Array<{
    id: string;
    tier: 'platform' | 'integration' | 'workspace' | 'behavioral';
    level: SkillCondensationLevel;
    tokens: number;
  }>;
  totalSkillTokens: number;
  budgetAllocated: number;
}
```

This is a JSONB column, not separate tables — keeps the logging path fast and avoids schema changes when new skill types are added.

## 9. Skill Improvement Pipeline

Covers The Five-Step Cycle, Automation Level by Step.
Touches `ai_usage_log`, `skill_context` tables.

### The Five-Step Cycle

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  1. Collect  │────▶│  2. Detect   │────▶│  3. Draft    │────▶│  4. Validate │────▶│  5. Approve  │
│  Usage Data  │     │  Patterns    │     │  Updates     │     │  via Evals   │     │  & Deploy    │
│              │     │              │     │              │     │              │     │              │
│  Automatic   │     │  Automatic   │     │  Agent or    │     │  Automatic   │     │  Human       │
│              │     │              │     │  Human       │     │              │     │              │
└─────────────┘     └─────────────┘     └─────────────┘     └─────────────┘     └─────────────┘
```

**Step 1 — Data Collection (fully automatic):** Every AI call logs skill context, token usage, and user action to `ai_usage_log`. No human involvement. Already specced in ai-metering.md — the only addition is the `skill_context` JSONB column.

**Step 2 — Pattern Detection (fully automatic):** A scheduled BullMQ job (weekly cadence) analyzes usage data. Produces structured findings — not vague alerts, but specific reports:

```
Finding: automation-builder skill, "Scheduled trigger" section
- Regeneration rate: 23% (up from 8% last week)
- Root cause: 14 of 19 failures produced 3-field cron expressions; EveryStack uses 5-field
- Affected task type: generate_automation
- Confidence: high (clear pattern in failure examples)
```

For integration skills, an additional trigger: integration eval drift. The weekly eval run detects when a Google Analytics or HubSpot skill's test cases start failing because the external platform changed something.

**Step 3 — Draft Updates (agent-assisted or human):** Given a finding, produce an updated skill section. The Skill Maintenance Agent (see §10) handles this automatically for clear cases. Complex or ambiguous cases are flagged for human review. The agent can also investigate integration drift by running exploratory MCP tool calls against the live integration and comparing actual behavior to the skill's description.

**Step 4 — Eval Validation (fully automatic):** Run the eval suite against the updated skill. Compare pass rates to baseline. If improved with no regressions → clean update, ready for approval. If mixed results → flag for human judgment.

**Step 5 — Approve & Deploy (human required):** A skill update affects AI quality for every user in that skill's domain. The approval step is lightweight: review the diff (what changed), the eval delta (before/after pass rates), and the failure pattern that motivated the change. Approve → skill is live. Reject → feedback to agent or manual edit.

### Automation Level by Step

| Step | Automation Level | Human Time (per cycle) |
|------|-----------------|----------------------|
| Data collection | 100% automatic | 0 |
| Pattern detection | 100% automatic | 0 |
| Draft updates | ~80% automatic (agent), 20% human for edge cases | 15–30 min reviewing agent drafts |
| Eval validation | 100% automatic | 0 |
| Approve & deploy | Human review, agent-prepared | 15–30 min reviewing diffs + results |

**Without the agent:** ~4–6 hours/week of manual log review, analysis, editing, eval running.
**With the agent:** ~30–60 minutes/week of reviewing and approving agent proposals.

## 10. Skill Maintenance Agent

Covers Agent Type, Scope, Trigger & Cadence, Execution Flow, Approval Model, Presentation in Platform Owner Console.
Touches `ai_usage_log` tables. See `agent-architecture.md`.

### Agent Type

A **platform-level agent** operating on behalf of the platform owner — not individual workspace users. Joins the post-MVP agent type catalog alongside Report Builder, Data Cleanup, etc. (see `agent-architecture.md`).

### Scope

Narrow and well-defined:

| Can Do | Cannot Do |
|--------|-----------|
| Read `ai_usage_log` (anonymized) | Access user workspace data |
| Read eval suite results | Deploy skills to production |
| Read current skill documents | Modify the AI pipeline code |
| Read reference docs (`docs/reference/`) | Modify tool definitions or provider config |
| Write draft skill updates | Approve its own changes |
| Run eval suite against drafts | Access metering or billing data |
| Run exploratory MCP tool calls (for integration drift investigation) | Make outbound API calls beyond MCP |
| Package updates as PRs with summaries | Modify any code outside `skills/documents/` |

### Trigger & Cadence

- **Scheduled:** Weekly (Monday, configurable). Runs the full cycle: analyze → detect → draft → eval → package.
- **On-demand:** Triggered from Platform Owner Console when a skill is flagged or when a new integration is added.
- **Integration drift:** Triggered when the weekly eval run detects regression in integration skill test cases.

### Execution Flow

```
1. Query ai_usage_log for past week
   → Aggregate: regeneration rates, failure patterns, token efficiency per skill

2. Run eval suite against all current skills
   → Establish baseline pass rates

3. For each skill with declining metrics:
   a. Pull failed examples from usage log
   b. Identify root cause (pattern matching on failure types)
   c. For integration skills with eval drift:
      - Run exploratory MCP tool calls against live integration
      - Compare actual behavior to skill description
      - Identify what changed
   d. Load the current skill document + relevant reference docs
   e. Draft an updated section addressing the root cause
   f. Run eval suite against the draft
   g. Compare to baseline

4. Package results:
   - For clean improvements (pass rate up, no regressions):
     → Create a pending update with diff, eval delta, and failure examples
   - For mixed results:
     → Flag for human review with analysis and recommendation
   - For integration drift:
     → Include the specific behavioral changes detected

5. Notify platform owner via Platform Owner Console notification
   → Summary: "3 skill updates proposed, 1 flagged for review"
```

### Approval Model

Uses the existing tiered approval framework from `agent-architecture.md`:

- **Skill updates with clean eval improvements:** Medium-risk tier → requires explicit human confirmation, but low-friction (one-click approve with diff visible)
- **Skill updates with mixed eval results:** High-risk tier → requires review of the trade-off analysis before approval
- **New integration skills (first draft):** High-risk tier → human reviews the full skill content
- **Emergency skill disable (eval pass rate below 50%):** The agent can recommend disabling a skill (falling back to no-skill behavior for that domain) pending review. It cannot disable automatically.

### Presentation in Platform Owner Console

The agent's output appears in Settings → AI → Skill Performance → Pending Updates:

Each pending update shows:
- Skill name and section changed
- The specific failure pattern that motivated the update
- Diff view (current → proposed)
- Eval results: before and after, per assertion
- Example failures that would now succeed
- Approve / Reject / Edit buttons

This is the same 30-minute weekly review surface described in §9. Most weeks, you scan 2–3 clean updates, approve them, and move on.

## 14. Key Architectural Decisions

**Tier 1 skills are static, versioned, human-written (or agent-drafted, human-approved) documents.** Platform feature knowledge doesn't change per session. Writing skills once and maintaining them through the improvement pipeline is dramatically more efficient than the AI rediscovering the platform every session.

**Tier 2 and Tier 3 skills are dynamically generated.** Workspace configuration and usage patterns change over time. These skills are generated from queries against existing tables, cached like SDS output, and invalidated when the underlying configuration changes.

**Integration skills include version awareness.** External platforms change without asking. The `lastVerified` date and `externalApiVersion` metadata, combined with automated eval drift detection, ensures stale skills get flagged and updated rather than silently degrading quality.

**The Skill Maintenance Agent drafts but never deploys.** Skill changes affect AI quality for every user in that domain. Human approval is always required. The agent eliminates the tedious analysis and drafting work; the human provides judgment on whether the change should ship.

**The AI never makes direct external API calls.** External communication flows through sync adapters, MCP, and webhooks. API knowledge skills give the AI reference knowledge to configure these channels correctly — the AI understands APIs without calling them directly.
